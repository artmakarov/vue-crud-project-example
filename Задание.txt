Задание:
CRUD-таблица на Vue 3 + Pinia + composable + fake API

Цель:
Сделать небольшой модуль “Справочник” — CRUD для сущности в таблице: список → создание → редактирование → удаление. Обязательно использовать composable, Pinia, и fake API.


1) Данные (сущность)

Сущность: Applicant (или School, на выбор — но лучше Applicant).

Поля (минимум):
id: number
fullName: string (обязательное)
phone: string (маска/валидация — достаточно простой)
status: 'new' | 'in_work' | 'done'
createdAt: string (ISO)


2) Функциональность

Таблица (list)

Пагинация (можно “клиентскую”, можно “серверную” — как в fake API удобнее).

Поиск по fullName (debounce приветствуется).

Фильтр по status.

Сортировка по createdAt или fullName.

Отображение состояний: loading / empty / error.

Создание / редактирование

Форма в диалоге (Vuetify), валидация обязательных полей.

Сохранение через store → API.

Обработка ошибок (snackbar/toast).

Удаление

Подтверждение удаления (dialog).

После удаления — обновление таблицы без перезагрузки страницы.

Реальное время (минимально)

Имитация “live update”: например, таймером раз в N секунд добавлять/обновлять запись через fake API или локально (или через простой WebSocket mock). Главное — показать, что умеешь обновлять данные реактивно и аккуратно.


3) Архитектурные требования (обязательные)

Pinia store: состояние списка, параметры (page/search/filter/sort), загрузка, ошибки.

Composable useApplicants() (или аналог): инкапсулирует работу со store + параметры + методы fetch/create/update/remove.

Fake API:

Вариант А (предпочтительно): MSW (у вас он уже есть в проекте) — GET/POST/PUT/DELETE.

Вариант Б: json-server.

Вариант В: локальный мок-сервис с setTimeout и “псевдо-DB” в памяти.

Код по eslint-стайлу (как минимум без явных нарушений/хака).

Чистота кода и структура (store/composable/components).
